## What the product actually does

It is a reference product where the user interacts with a large body of structured content:

- classes;
- species;
- feature docs;
- related reference materials;
- user-facing flows that already go beyond a static encyclopedia.

That makes the real topic here the balance between content, rendering, data, and delivery.

## Architecture map

| Layer | What it does | Why it matters |
| --- | --- | --- |
| Frontend | React 19, TypeScript, Vite SSR, routing, search, UI state | Provides the product interface for navigating a large reference catalog |
| Data layer | TanStack Query, caching, revalidation, repeated requests | Keeps UX stable where data is large and state should not turn chaotic |
| Content layer | Knowledge sections, feature docs, a dedicated content path | Lets reference content evolve as content, not only as UI implementation |
| Backend | Django, DRF, JWT, docs, comments/profiles/notifications | Separates auth and system flows into their own contractual layer |

## Data, state, and interaction

One of the key problems in a product like this is not merely showing data, but keeping transitions predictable while
the user moves through large content surfaces. The guide should not flicker, reset, or feel like every page load starts
from zero.

```ts
export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000,
      retry: 1,
      refetchOnWindowFocus: true,
    },
  },
});
```

This is not a magical fix, but it is the right baseline for a large reference product where query behavior needs to
stay calm and predictable.

## Content and knowledge sections

A large share of the product is reference content. That is why knowledge sections now have their own development path,
so they can behave like a content system instead of being trapped inside UI components.

Already present in that layer:

- public routes for `classes` and `species`;
- a dedicated `ClassesFeatures` tree;
- manifest and fallback logic for feature docs;
- a separate delivery path for reference pages.

```ts
const featureModules = import.meta.glob(
  "../../../../dnd/src/content/ClassesFeatures/**/*.mdx"
);

const manifestModules = {
  ...import.meta.glob("../../../../dnd/src/content/ClassesFeatures/manifest.ts"),
  ...import.meta.glob("../../../../dnd/src/content/ClassesFeatures/manifest.js"),
  ...import.meta.glob("../../../../dnd/src/content/ClassesFeatures/manifest.json"),
};
```

The point is not the `glob` call itself. The point is that content gets its own structure instead of remaining an
accidental side effect of components.

## Public routes and a separate delivery path

Knowledge sections already use a separate delivery path where public pages can be served without unnecessary client
noise, and interactive parts can be attached only where they are justified.

```astro
---
import ClassSearchIsland from "../../components/ClassSearchIsland.tsx";
---

<ClassSearchIsland client:visible items={classSearchItems} />
```

That split keeps reference pages lighter and reserves interactivity for the places where it actually matters.

## Backend and system flows

The backend layer is not decorative. It handles:

| Module | What it covers |
| --- | --- |
| Django REST Framework | REST contract for the client |
| Simple JWT | Authentication and protected user flows |
| OpenAPI / docs | Transparent endpoint validation and integration |
| Docker + Fly.io flow | Repeatable delivery for the live API |
| Env-driven settings | Controlled configuration across local/dev/prod |

That gives the product proper responsibility boundaries instead of one repo-shaped blob.

## Active development zones

There are still areas that live outside the public knowledge sections, and they should:

- auth;
- profiles;
- comments;
- notifications;
- feed / posts.

They demand a different level of interactivity than reference pages, so they do not belong in the same delivery pattern.

## Current product state

Right now the product already has:

1. a large knowledge catalog with its own structure;
2. a client UI with real data behavior;
3. a content layer that is separate from auth and user flows;
4. a product that can evolve in slices instead of one large rewrite.
