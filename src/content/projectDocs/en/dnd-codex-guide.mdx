## Fast read

`DND Codex Guide` started as a large interactive guide on React, but over time it became clear that the public
encyclopedic surface has a different nature than auth, comments, notifications, or user flows.

That is why the project is not moving toward “more client-side JS everywhere.” It is moving toward a
<strong>controlled split of responsibilities</strong>:

- React/Vite SSR remains where rich interaction, state, forms, and product flows are needed.
- Astro takes over public knowledge routes where HTML-first delivery and lower hydration cost matter more.
- MDX becomes the editorial format for structured feature documentation.

## Architecture model

| Layer | What lives there | Why it exists |
| --- | --- | --- |
| React 19 + Vite SSR | Main product UI, rich interaction, forms, client flows, routing, caching | Keeps the application flexible where a page is more than just a document |
| TanStack Query | Server state, revalidation, repeated requests, stable transitions | Makes data behavior predictable and avoids UI flicker |
| Astro pilot | `classes`, `species`, static public routes, knowledge pages | Reduces hydration scope and restores HTML-first delivery for public sections |
| MDX content tree | Class features, editorial notes, structured docs | Gives content its own maintainable shape instead of burying it in JS blobs |
| Django backend | Auth, API, comments, profiles, notifications, OpenAPI docs | Separates data contracts and user-data concerns from the UI layer |

## Astro is used as a migration tool, not as decoration

A separate `dnd-astro-pilot` already exists as a working proof of concept. It includes both `@astrojs/react` and
`@astrojs/mdx`, plus a legacy alias into the current app so public sections can move gradually instead of forcing a
total rewrite.

```js
import { defineConfig } from "astro/config";
import mdx from "@astrojs/mdx";
import react from "@astrojs/react";

export default defineConfig({
  integrations: [mdx(), react()],
});
```

The important point is simple: <strong>Astro is not wrapping the whole old frontend</strong>. It is becoming a cleaner
delivery path for the public knowledge surface.

## MDX pipeline for class features

The most interesting upgrade is not “another layout.” It is the fact that class features already live in an MDX tree.
That turns heavy reference content into an editorial system instead of a UI-bound implementation detail.

| Mechanic | How it works | Why it matters |
| --- | --- | --- |
| `ClassesFeatures/**/*.mdx` | Each feature becomes its own content module | Feature docs stop being trapped inside one large JS layer |
| `manifest.ts/js/json` | Can define ordering, grouping, and bindings | Ordering no longer needs to be hardcoded into page logic |
| Fallback scanning | If no manifest is present yet, the route scans the tree | Migration keeps moving even when content structure is still incomplete |
| `import.meta.glob()` | Loads feature modules as a content tree | It is a clean content-loading mechanism, not a custom improvised loader |

```ts
const featureModules = import.meta.glob(
  "../../../../dnd/src/content/ClassesFeatures/**/*.mdx"
);

const manifestModules = {
  ...import.meta.glob("../../../../dnd/src/content/ClassesFeatures/manifest.ts"),
  ...import.meta.glob("../../../../dnd/src/content/ClassesFeatures/manifest.js"),
  ...import.meta.glob("../../../../dnd/src/content/ClassesFeatures/manifest.json"),
};
```

The value here is not the `glob` itself. The value is that content becomes <strong>operationally manageable</strong>.

## Selective hydration instead of “hydrate everything”

React does not disappear in the pilot, but it also does not mount without reason. Search lives in islands, which means
public knowledge pages stop paying for a full client runtime up front.

```astro
---
import ClassSearchIsland from "../../components/ClassSearchIsland.tsx";
---

<ClassSearchIsland client:visible items={classSearchItems} />
```

That is a small line of code with a large implication: a content page stays a page instead of becoming a full SPA shell
for every paragraph.

## The main frontend is still strong on its own

Moving public slices into Astro does not mean the React core has become obsolete. The live frontend already carries the
stack needed for a serious product UI:

- `React 19`
- `Vite SSR + prerender`
- `TanStack Query`
- `React Router 7`
- `React Hook Form + Zod`
- `Framer Motion`
- `vite-plugin-pwa`
- `react-markdown + remark-gfm + rehype-raw`

The data layer is also configured with mature defaults instead of a noisy refetch loop:

```ts
export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000,
      retry: 1,
      refetchOnWindowFocus: true,
    },
  },
});
```

So the React layer here is not something to hide. It remains the right tool where product complexity actually demands it.

## Backend as a separate infrastructure contour

The Django API already lives as its own layer:

| Module | What it covers |
| --- | --- |
| Django REST Framework | REST contract for the client |
| Simple JWT | Authentication and protected user flows |
| OpenAPI / docs | Clear integration and quick endpoint validation |
| Docker + Fly.io flow | Repeatable deployment for the live API |
| Env-driven settings | Controlled configuration across local/dev/prod |

That matters because `DND Codex` is no longer a single UI repo in practice. It is already a system with several related,
but distinct, layers.

## Why this case deserves a dedicated page

Because it shows more than a final UI. It shows <strong>how a system is evolved without a fake hero refactor</strong>.
There is:

1. a live React product;
2. a dedicated Astro layer for public knowledge delivery;
3. an MDX pipeline for editorial content;
4. a backend that remains the contract surface for data and user flows.

That is what makes `DND Codex Guide` interesting in the portfolio right now: not because it is “the main project,” but
