## Що саме робить продукт

Це не демка однієї анімації і не презентація одного технічного апдейту. Це довідковий продукт, у якому користувач
постійно працює з великим набором даних:

- класи;
- види;
- feature docs;
- пов'язані довідкові матеріали;
- користувацькі сценарії, які вже виходять за межі простої статичної енциклопедії.

Тому тут важлива не одна бібліотека сама по собі, а баланс між контентом, рендером, даними й delivery.

## Дані, стани й взаємодія

Одна з ключових задач у такому продукті це не просто “показати дані”, а тримати переходи між сторінками й запитами
передбачуваними. Тут важливо, щоб довідник не мерехтів, не ламав темп читання і не перетворював кожне відкриття
сторінки на новий холодний старт.

```ts
export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 5 * 60 * 1000,
      retry: 1,
      refetchOnWindowFocus: true,
    },
  },
});
```

Це не “магічний оптимізатор”, але це правильна база для великого knowledge-продукту, де запити мають поводитися
спокійно і передбачувано.

## Контент і knowledge-секції

Окремий великий шмат цього продукту це довідковий контент. Саме тому knowledge-секції розвиваються окремим шляхом,
щоб з ними можна було працювати як з контентною системою, а не тільки як з UI-компонентами.

Що вже є в цьому контурі:

- публічні маршрути для `classes` і `species`;
- окреме дерево `ClassesFeatures`;
- manifest/fallback-механіка для feature docs;
- окремий delivery-path для довідкових сторінок.

```ts
const featureModules = import.meta.glob(
  "../../../../dnd/src/content/ClassesFeatures/**/*.mdx"
);

const manifestModules = {
  ...import.meta.glob("../../../../dnd/src/content/ClassesFeatures/manifest.ts"),
  ...import.meta.glob("../../../../dnd/src/content/ClassesFeatures/manifest.js"),
  ...import.meta.glob("../../../../dnd/src/content/ClassesFeatures/manifest.json"),
};
```

Тут суть не в самому `glob`, а в тому, що контент отримує окрему структуру і перестає бути випадковим придатком
компонентів.

## Публічні сторінки і окремий delivery-path

Для knowledge-секцій уже піднятий окремий delivery-path, де публічні сторінки можна віддавати без зайвого клієнтського
шуму, а interactive частини підключати точково.

```astro
---
import ClassSearchIsland from "../../components/ClassSearchIsland.tsx";
---

<ClassSearchIsland client:visible items={classSearchItems} />
```

Це важливо не тому, що тут є конкретно Astro. Це важливо тому, що продукт розділяється по ролях: довідковий контент,
пошук, UI і user flows перестають жити одним безформним шаром.

## Backend і системні сценарії

Окремий backend-контур тут не декоративний. Саме він закриває:

| Module | Що закриває |
| --- | --- |
| Django REST Framework | REST-контракт для клієнта |
| Simple JWT | Авторизацію і захищені user flows |
| OpenAPI / docs | Прозору перевірку endpoint-ів і інтеграцію |
| Docker + Fly.io flow | Повторюваний delivery для live API |
| Env-driven settings | Керований конфіг для local/dev/prod |

Це дає продукту межі відповідальності, а не один repo-shaped клубок.

## Активні зони розвитку

Тут досі є шматки, які живуть окремо від публічних knowledge-секцій і правильно, що так:

- auth;
- profiles;
- comments;
- notifications;
- feed / posts.

Усі вони вимагають іншого рівня інтерактивності, ніж довідкові сторінки, тому й не змішуються в один delivery-патерн.

## Чому цей кейс важливий

Бо він показує продукт не як набір окремих технологій, а як систему, у якій:

1. великий knowledge-каталог має свою структуру;
2. клієнтський UI має свою логіку роботи з даними;
3. контент і рендер відокремлюються від auth/user flows;
4. продукт можна розвивати шарами, а не одним великим переписуванням.
